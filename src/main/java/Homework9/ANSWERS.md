1. **Нужно ли было в принципе писать свой собственный генератор с фамилиями? Было бы проще использовать генератор из datafaker?**
    Да, писать собственный генератор фамилий не требуется.
   Библиотека datafaker уже содержит готовый метод faker.name().lastName(), который генерирует случайные фамилии из большого набора реальных английских (и, при смене локали, национальных) фамилий. Использование готового генератора упрощает код, уменьшает вероятность ошибок и делает тесты более репрезентативными, поскольку набор фамилий гораздо шире, чем несколько вручную заданных строк.

2. **Какая из двух библиотек из этого задания может больше пригодиться при написании автотестов – jfiglet или datafaker?**
   Для автотестов более полезной будет datafaker.

- datafaker позволяет генерировать разнообразные тестовые данные (имена, адреса, номера телефонов, даты, номера кредитных карт и т.д.), что является типичной задачей при подготовке входных параметров тестов.
- jfiglet служит лишь для визуального представления текста в виде ASCII‑арт; в контексте автоматизированного тестирования она не предоставляет функциональности, влияющей на проверку бизнес‑логики или UI.

3. **Каким образом JAR‑файлы этих библиотек попали к нам в проект?**
   JAR‑файлы попали в проект через систему управления зависимостями Gradle.
   В файле build.gradle в секции dependencies были указаны строки:

`implementation 'net.datafaker:datafaker:2.5.4'`

`implementation 'com.github.lalyos:jfiglet:0.0.8' `

После сохранения файла Gradle выполнил резолвинг зависимостей: он скачал соответствующие артефакты из репозитория Maven Central (https://repo1.maven.org) и разместил их в локальном кэше Gradle (~/.gradle/caches). При сборке проекта Gradle автоматически добавил эти JAR‑файлы в classpath, что сделало их доступными для компиляции и выполнения кода.
